# Case Study: HPLC Data Visualizer

## Summary

HPLC Data Visualizer is a lightweight research-software project built around a real workflow in a pharmaceutical sciences / natural products research lab. The project was designed to reduce the time and friction involved in preparing HPLC chromatogram figures for lab meetings, presentations, and research discussions.

Rather than positioning the project as a broad open-source library, this case study frames it as proof of work: identifying a specific scientific workflow pain point, building a focused browser-based tool, deploying it, collecting user feedback, and iterating based on real usage.

## Context

I am a PhD student in pharmaceutical sciences / natural products research. In our lab, HPLC chromatogram figures are frequently prepared for internal discussions, group meetings, and presentation slides.

The traditional workflow often depends on Origin or similar desktop software:

1. Export raw HPLC data from the instrument software.
2. Import one or more files into plotting software.
3. Create or adjust layers for multi-sample comparison.
4. Tune axes, line colors, curve order, and labels.
5. Mark peaks or retention times when needed.
6. Export a figure.
7. Repeat formatting adjustments to keep figures visually consistent.

For a typical presentation-ready figure set, especially when comparing multiple samples, this can take around 15-30 minutes. The work is repetitive and often happens under time pressure before meetings or research discussions.

## Problem

The pain point was not that HPLC plotting was impossible. The problem was that the repeated preparation of clean, consistent chromatogram figures required too many manual steps.

Common friction points included:

- Repeated data import and formatting in desktop plotting software.
- Inconsistent styles between figures prepared at different times.
- Multi-sample comparisons requiring manual layer and axis adjustment.
- Legends becoming hard to read when several chromatograms are shown together.
- SVG or presentation-ready export being available but not always obvious to new users.
- New lab members needing time to learn the formatting workflow before producing usable figures.

The target user was not a general public audience. The target user was a lab member who already has HPLC data and needs a clean figure quickly.

## Solution

I built a Streamlit web app that lets users upload HPLC data and generate standardized chromatogram figures through a browser interface.

The core product idea was simple:

- Keep the workflow narrow.
- Make the default output clean enough for discussion slides.
- Provide only the controls that matter for HPLC figure preparation.
- Reduce formatting decisions that users repeat manually.
- Make export obvious.

The intended workflow is:

1. Upload HPLC data, or load built-in example data.
2. Compare multiple chromatograms in a stacked view.
3. Adjust styling from a sidebar.
4. Label samples directly beside curves.
5. Mark retention times or highlight peak regions when useful.
6. Export the result as an SVG figure.

## Product Decisions

Several features came directly from observing or experiencing the lab workflow:

- **Direct curve labels:** Color-only legends were hard to read for stacked multi-sample plots, so sample names can be shown at the left or right edge of each curve.
- **Adjustable label size:** Presentation figures need readable labels, so sample-name font size is adjustable.
- **Hidden legend and Y-axis by default:** Many presentation chromatograms are cleaner without extra chart furniture.
- **Color palettes:** Users can switch between built-in palettes for different presentation needs, including more restrained or colorblind-safe options.
- **Visible SVG export button:** Plotly already includes export through the toolbar, but new users can miss it. A dedicated download button makes the export path explicit.
- **Built-in example data:** New users can try the app immediately without preparing files first.

## Real-World Use

The app has been used by members of my lab. Figures generated by the app have been used in group meetings and research presentations.

Current evidence:

- Lab usage: used by members of my lab.
- Presentation usage: figures generated by the app have been used in group meetings and research presentations.
- External evaluation: a postdoctoral researcher and a principal investigator from two other universities tested the app independently.
- External feedback: both supported the core product idea. The built-in example workflow produced useful results, and the remaining changes were described as relatively minor.

The feedback showed that the product concept and current workflow were understandable and useful beyond the lab where the project originated.

The most relevant success signal is not public traffic or GitHub stars. This is a highly vertical tool. The more meaningful validation is whether the tool is used by its intended users and whether it replaces a painful manual workflow.

My responsibilities included:

- Identifying the workflow pain point from real lab practice.
- Defining the target user and minimum useful workflow.
- Designing the browser-based interaction model.
- Implementing file parsing, plotting, stacking, labeling, marking, region highlighting, and SVG export.
- Deploying the app for browser-based access.
- Collecting feedback from lab users.

This project reflects both domain understanding and software execution: I was not solving an abstract plotting problem, but a specific workflow problem I had seen repeatedly in a research environment.

## Development Workflow

This project was developed with LLM-assisted coding.

I used AI coding tools to accelerate implementation, explore Streamlit and Plotly interaction patterns, debug issues, and quickly iterate on UI details. The domain framing, product decisions, validation criteria, and user feedback loop came from my own research environment.

The workflow was:

1. Identify a repeated manual task in the lab.
2. Build a minimal working prototype.
3. Test it with realistic HPLC data.
4. Use lab feedback to identify usability issues.
5. Iterate on controls, defaults, labeling, and export behavior.
6. Test the workflow with researchers from other institutions and adapt the parser to real instrument exports.
7. Keep the tool lightweight enough to remain easy to deploy and maintain.

Examples of iteration:

- Added direct sample labels after realizing the legend was not enough for multi-sample plots.
- Moved sample labels outside the plot area so they would not cover chromatogram curves.
- Added label font-size control after presentation readability feedback.
- Changed default display settings to reduce visual clutter.
- Added a visible SVG export button after noticing the toolbar export was too easy to miss.
- Added built-in example data to make the first-use experience easier.
- Added multi-signal CSV selection, import diagnostics, and CTX parsing after testing exports from instruments used outside my lab.
